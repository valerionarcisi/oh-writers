# Spec 18 — Internationalization (i18n)

## Overview

Oh Writers supports multiple languages for the UI. Content (screenplay text, loglines, etc.) stays in whatever language the writer uses — only the interface labels, buttons, menus, and system messages are translated.

---

## Languages

| Phase  | Languages                                 |
| ------ | ----------------------------------------- |
| Launch | English (default), Italian                |
| Later  | French, Spanish, German (based on demand) |

---

## What Gets Translated

- Navigation labels (Projects, Overview, Logline, Synopsis, Outline, Treatment, Screenplay, etc.)
- Button text (Save, Export, New project, Sign in, etc.)
- Form labels and placeholders
- Error messages (Document not found, Invalid email, etc.)
- Document type names (Logline → Logline, Outline → Scaletta in Italian)
- Status text (Saved, Saving..., soon, etc.)
- Date/time formatting (Apr 13 vs 13 apr)

## What Does NOT Get Translated

- User content (screenplay text, loglines, synopses)
- Fountain format keywords (INT., EXT., FADE IN — these are industry standard in English)
- Character names, scene headings
- API responses, technical error details

---

## Technical Approach

### Library: none (simple key-value)

For a small set of languages and ~200-300 keys, a full i18n library (i18next, react-intl) is overkill. A simple module with typed keys is enough.

```typescript
export type TranslationKey = keyof (typeof translations)["en"];
```

### Hook

```typescript
// apps/web/app/hooks/useTranslation.ts
const useTranslation = () => {
  const locale = useLocale(); // from context or user preference
  const t = (key: TranslationKey): string => translations[locale][key];
  return { t, locale };
};
```

### Locale Detection

1. User preference (stored in `users.locale` column) — takes priority
2. Browser `navigator.language` — fallback
3. English — default

### Date/Number Formatting

Use `Intl.DateTimeFormat` and `Intl.NumberFormat` with the current locale. No library needed.

```typescript
const formatDate = (date: Date, locale: Locale) =>
  new Intl.DateTimeFormat(locale, { month: "short", day: "numeric" }).format(
    date,
  );
```

---

## DB Change

Add `locale` column to `users` table:

```sql
ALTER TABLE users ADD COLUMN locale text NOT NULL DEFAULT 'en';
```

---

## Implementation Order

1. Create translation keys file in `packages/domain`
2. Add `useTranslation` hook
3. Add locale context provider
4. Replace hardcoded strings in sidebar, toolbar, forms
5. Add locale selector in user settings
6. Add `locale` column to users table

---

## Not In Scope

- RTL languages (Arabic, Hebrew) — would require layout changes
- Pluralization rules — keep messages simple to avoid needing ICU
- Translation management platform (Crowdin, Phrase) — overkill until 5+ languages
- Server-side translated error messages — errors stay in English, only UI labels are translated
