# 13 — Locations

## Overview

The locations module connects the fictional places in the screenplay to real, physical places that can be filmed. It supports the full scouting workflow: from the first list of candidates generated by AI, through photo uploads and notes during scouting, to the final confirmation of each location.

Locations feed directly into the schedule (grouping scenes by location) and the budget (location fees per day).

---

## Core Concepts

### Location Requirement vs Location Candidate

These are two distinct things:

- **Location Requirement** — a fictional place extracted from the breakdown (e.g. "MARCO'S APARTMENT", "ROOFTOP AT NIGHT", "COUNTRYSIDE ROAD"). One per unique location in the screenplay.
- **Location Candidate** — a real place being considered for a requirement. One requirement can have multiple candidates. The production picks one.

```
Location Requirement: "ABANDONED FACTORY"
├── Candidate A: Ex-Metalfaber, Via Industriale 14, Milano
├── Candidate B: Ex-Ceramiche Pozzi, Vimercate
└── Candidate C: Studio set (build from scratch)  ← confirmed ✓
```

### Location Status Flow

```
requirement created (from breakdown)
  → candidates added (manually or AI suggested)
    → scouting in progress (photos, notes, permits research)
      → candidate confirmed
        → location locked (in schedule)
```

## Data Model

```typescript
export const LocationRequirementSchema = z.object({
  id: z.string().uuid(),
  projectId: z.string().uuid(),
  name: z.string().min(1).max(200), // as it appears in the screenplay
  description: z.string().nullable(),
  intExt: z.enum(["INT", "EXT", "INT/EXT"]),
  timeOfDay: z.array(z.enum(["DAY", "NIGHT", "DAWN", "DUSK"])),
  sceneIds: z.array(z.string().uuid()), // scenes that use this location
  confirmedCandidateId: z.string().uuid().nullable(),
  status: z.enum(["pending", "scouting", "confirmed", "locked"]),
});

export const LocationCandidateSchema = z.object({
  id: z.string().uuid(),
  requirementId: z.string().uuid(),
  name: z.string().min(1).max(200), // real name of the place
  address: z.string().nullable(),
  coordinates: z
    .object({
      lat: z.number(),
      lng: z.number(),
    })
    .nullable(),
  contactName: z.string().nullable(),
  contactEmail: z.string().email().nullable(),
  contactPhone: z.string().nullable(),
  estimatedDailyFee: z.number().nullable(), // feeds budget
  permitRequired: z.boolean().nullable(),
  permitNotes: z.string().nullable(),
  availableFrom: z.date().nullable(),
  availableTo: z.date().nullable(),
  notes: z.string().nullable(),
  photos: z.array(LocationPhotoSchema),
  status: z.enum(["candidate", "visited", "rejected", "confirmed"]),
  aiSuggested: z.boolean().default(false),
});

export const LocationPhotoSchema = z.object({
  id: z.string().uuid(),
  candidateId: z.string().uuid(),
  url: z.string().url(),
  caption: z.string().nullable(),
  uploadedAt: z.date(),
});

export type LocationRequirement = z.infer<typeof LocationRequirementSchema>;
export type LocationCandidate = z.infer<typeof LocationCandidateSchema>;
```

## AI Location Suggestions

### What the AI does

For each location requirement, the AI suggests real candidate locations based on:

- The fictional description in the screenplay (name, visual characteristics, scene context)
- Project country/region (set during project creation)
- Format and genre (a horror film has different location needs than a romantic comedy)

The AI returns candidate names, general area, and reasoning. It does not return precise addresses or contact information — that requires human research.

### What the AI does NOT do

- Find actual permit information (too jurisdiction-specific)
- Provide rental prices (too variable)
- Replace a location scout

AI suggestions are starting points for the human scout, not decisions.

```typescript
export const AiLocationSuggestionSchema = z.object({
  requirementId: z.string().uuid(),
  suggestions: z.array(
    z.object({
      name: z.string(),
      area: z.string(), // city or region
      reasoning: z.string(), // why this place fits the scene
      searchQuery: z.string(), // suggested Google Maps / web search query
      confidence: z.number().min(0).max(1),
    }),
  ),
});
```

## User Flows

### Location requirements list

1. Generated automatically from confirmed breakdown
2. User sees all unique locations in the screenplay
3. Each row shows: location name, INT/EXT, time of day, number of scenes, status
4. Filter by status (pending, scouting, confirmed)

### Add candidates

1. User opens a requirement
2. "Get AI suggestions" → AI returns 3–5 candidates with reasoning
3. User can accept a suggestion (creates a candidate), reject it, or add a candidate manually
4. Each candidate has its own detail view

### Scout a location

1. User visits the candidate's detail view
2. Adds: address, coordinates (via map picker), contact info, fee estimate, permit notes
3. Uploads photos (multiple, with captions)
4. Sets status to `visited`

### Confirm a location

1. From the requirement view, user selects one candidate and confirms it
2. Requirement status → `confirmed`
3. Confirmed candidate's `estimatedDailyFee` is synced to the budget location line
4. Schedule uses `confirmedCandidateId` to group scenes by location

### Map view

All confirmed candidates shown on a map. Useful for:

- Identifying shooting day clusters (nearby locations = efficient days)
- Sharing with the AD for schedule optimization

## Location Groups for Scheduling

When the schedule is generated, scenes are grouped by their confirmed location. The scheduler uses `confirmedCandidateId` and `coordinates` to minimize travel between locations.

```typescript
// Used by the scheduler to cluster scenes
export const LocationGroupSchema = z.object({
  requirementId: z.string().uuid(),
  confirmedCandidateId: z.string().uuid(),
  coordinates: z.object({ lat: z.number(), lng: z.number() }).nullable(),
  sceneIds: z.array(z.string().uuid()),
  estimatedDailyFee: z.number().nullable(),
});
```
